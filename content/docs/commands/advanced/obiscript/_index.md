## Description

{{< obi obiscript >}} lets a biologist apply fully custom logic to every sequence in
a dataset by writing a short [Lua](https://www.lua.org/) script, without recompiling
{{% obitools4 %}} or writing Go code. It is the right tool whenever the built-in
commands — {{< obi obigrep >}}, {{< obi obiannotate >}}, and the like — are not
flexible enough: for instance to compute a new attribute from the sequence itself,
to maintain a running counter across all records, or to read an external table and
join it into the annotations.

The script is structured around three optional Lua functions. The `worker(sequence)`
function is the core: it is called once per input sequence record, receives the record
as a `BioSequence` object, and must return the (possibly modified) record — or `nil`
to drop it. The `begin()` function runs once before any record is processed and is
typically used for initialisation. The `finish()` function runs once after the last
record and is typically used to print summary statistics. A thread-safe key-value table
called `obicontext` is shared across all parallel worker invocations and can be used to
accumulate results safely.

An important behavioural detail: the sequence selection flags (such as `--min-length`,
`--predicate`, `--sequence`, etc.) do **not** filter sequences out of the output —
they select which sequences the `worker()` function is applied to. Sequences that do
not match the selection pass through to the output unchanged, without the script being
executed on them.

To bootstrap a new script, run {{< obi obiscript >}} with `--template`. It prints a
self-contained Lua skeleton with inline comments:

```bash
obiscript --template
```

The skeleton demonstrates all three lifecycle functions and shows how to use
`obicontext` for cross-record aggregation and `sequence:id()` to rename records.

## Options

#### {{< obi obiscript >}} specific options

- {{< cmd-option name="script" short="S" param="SCRIPT" >}}
  Path to the Lua script file to execute. The file must exist and be syntactically
  valid Lua. The script should define a `worker(sequence)` function, and optionally
  `begin()` and `finish()`.
  {{< /cmd-option >}}

- {{< cmd-option name="template" >}}
  Print a minimal Lua script template to standard output, with stubs for `begin()`,
  `worker()`, and `finish()` and inline documentation, then exit. Use this to
  bootstrap a new script.
  {{< /cmd-option >}}

- {{< cmd-option name="paired-mode" param="forward|reverse|and|or|andnot|xor" >}}
  When paired reads are provided, determines how filter conditions are applied to both
  reads of a pair. Default: `forward`.
  {{< /cmd-option >}}

#### Taxonomic options

- {{< cmd-options/taxonomy/taxonomy >}}

- {{< cmd-option name="restrict-to-taxon" short="r" param="TAXID" >}}
  Apply the script only to sequences whose taxid belongs to the specified taxon.
  {{< /cmd-option >}}

- {{< cmd-option name="ignore-taxon" short="i" param="TAXID" >}}
  Apply the script only to sequences whose taxid does NOT belong to the specified taxon.
  {{< /cmd-option >}}

- {{< cmd-option name="require-rank" param="RANK_NAME" >}}
  Apply the script only to sequences whose taxon has the specified rank (e.g.,
  `species`, `genus`).
  {{< /cmd-option >}}

- {{< cmd-option name="valid-taxid" >}}
  Apply the script only to sequences that carry a currently valid NCBI taxid.
  Default: `false`.
  {{< /cmd-option >}}

- {{< cmd-option name="fail-on-taxonomy" >}}
  Abort with an error if a taxid used during filtering is not currently valid.
  Default: `false`.
  {{< /cmd-option >}}

- {{< cmd-option name="update-taxid" >}}
  Automatically replace taxids declared as merged with their current equivalent.
  Default: `false`.
  {{< /cmd-option >}}

- {{< cmd-option name="raw-taxid" >}}
  Print taxids in output without supplementary information (taxon name and rank).
  Default: `false`.
  {{< /cmd-option >}}

- {{< cmd-option name="with-leaves" >}}
  When extracting taxonomy from a sequence file, attach sequences as leaves of their
  taxid annotation. Default: `false`.
  {{< /cmd-option >}}

## Examples

### Add a sample identifier to every sequence record

The file [sequences.fasta](sequences.fasta) contains six {{% fasta %}} sequences whose
identifiers follow the pattern `<sample>_<number>`. The script
[annotate.lua](annotate.lua) extracts the sample prefix and stores it as a new
attribute `sample` on each record using `sequence:attribute("sample", value)`.

{{< code "sequences.fasta" fasta true >}}

{{< code "annotate.lua" txt true >}}

```bash
obiscript --script annotate.lua --fasta-output -o annotated.fasta sequences.fasta
```

{{< code "annotated.fasta" fasta true >}}

### Apply a script selectively to sequences above a length threshold

The file [reads.fastq](reads.fastq) contains four {{% fastq %}} reads of varying
lengths. By combining {{< obi obiscript >}} with `--min-length`, the
[process_pairs.lua](process_pairs.lua) script is applied only to reads that are at
least 100 bp long; shorter reads pass through to the output without modification.

{{< code "reads.fastq" fastq true >}}

{{< code "process_pairs.lua" txt true >}}

```bash
obiscript --script process_pairs.lua --min-length 100 -o result.fastq reads.fastq
```

{{< code "result.fastq" fastq true >}}

### Enrich FASTQ records with a custom attribute and output as JSON

The [enrich.lua](enrich.lua) script adds a `processed` attribute to every sequence.
Combining this with `--json-output` produces a structured JSON array where each record
carries the new annotation alongside its sequence and quality data.

{{< code "sequences.fastq" fastq true >}}

{{< code "enrich.lua" txt true >}}

```bash
obiscript --script enrich.lua --json-output -o enriched.json sequences.fastq
```
